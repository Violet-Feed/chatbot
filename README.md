# Chatbot Orchestrator

一个面向群聊和 AI 私聊的多 Agent 编排系统。系统通过 RocketMQ 异步接收 IM 消息，用 Redis 做时间窗口聚合和发送调度，用 LangGraph 编排 LLM 决策、记忆更新与回复生成，并通过 MySQL 沉淀会话记忆。

## 业务架构

- 多 Agent 群聊：
    - 每个 Agent 有独立名称和人设，群聊中会根据当前话题选择更适合的 Agent 发言。
    - 同一群可以绑定多个 Agent，系统先加载候选 Agent，再由决策 LLM 判断是否需要回复以及由谁回复。
    - 回复生成会结合 Agent 人设、群聊信息、近期消息、历史消息和记忆，让发言贴近当前语境。
- AI 私聊：
    - 私聊会话通过 `con_id` 定位目标 Agent。
    - 私聊场景加载到 Agent 后直接生成回复，不额外做群聊发言竞争。
- 自动决策：
    - 消息先按会话进入时间窗口，窗口关闭后统一处理，避免逐条消息触发。
    - 群聊决策以“克制发言”为原则，只有在被点名、明确提问、话题强相关或需要澄清时才回复。
    - 决策结果包含是否回复、主回复 Agent 和触发原因，触发原因会继续传入生成环节。
- 拟人化策略：
    - 记忆驱动：短期摘要、长期摘要和黑话词表共同进入回复上下文。
    - 上下文驱动：优先参考最新窗口消息，必要时通过工具拉取更早历史消息。
    - 工具驱动：遇到实时信息或背景不足时，生成 Agent 可以调用搜索或上下文补齐工具。
    - 表达约束：Prompt 约束回复短、自然、贴近群聊语气，不自我介绍，不做报告式总结。

## 技术架构

### 数据流主链路

1. MQ 消息进入：`RocketMQ -> MessageConsumer -> WindowManager.on_message`
2. 窗口聚合：`WindowManager` 将有效文本消息写入 Redis 当前窗口缓冲
3. 窗口关闭：读取当前窗口和历史累计窗口，合并当前窗口到累计缓冲，并调用 `ConversationGraph.run`
4. LangGraph 编排：`load_context -> decide / update_memories -> generate -> send`
5. 回复调度：`send` 节点写入 Redis ZSET，`SendScheduler` 拉取到期任务并调用 `IMService.SendMessage`
6. 记忆更新：`update_memories` 节点并行更新短期摘要、长期摘要和黑话词表

### 多 Agent 决策如何实现

- `WindowManager` 只接收群聊和 AI 私聊中的文本消息，并补齐发送者名称。
- `ConversationGraph.load_context` 负责加载会话信息、候选 Agent、会话摘要和相关黑话：
    - 群聊通过 IM 服务获取会话内 Agent 列表，再通过 AIGC 服务加载 Agent 名称和人设。
    - AI 私聊从 `con_id` 解析 Agent id，再通过 AIGC 服务加载 Agent。
    - 会话摘要和黑话从 MySQL 读取，黑话会根据最近文本做相关性过滤。
- 群聊进入 `decide` 节点后，LLM 基于候选 Agent、当前窗口、最近历史、短期摘要、长期摘要、群名称和群描述输出结构化决策。
- 决策输出中的 Agent id 会在本地候选列表中校验，只有命中候选 Agent 才进入生成。
- AI 私聊直接选择当前私聊 Agent 进入生成。

### 自动说话如何实现

- 每条有效消息写入 `grp:{con_short_id}:win:buf`，并刷新窗口截止时间。
- `grp:{con_short_id}:win:close_scheduled` 作为窗口关闭任务锁，保证同一会话不会重复创建关闭任务。
- 窗口到期后，系统读取：
    - 当前窗口：`grp:{con_short_id}:win:buf`
    - 累计历史：`grp:{con_short_id}:win:all_buf`
- 当前窗口会合并进累计历史，随后删除当前窗口缓冲、窗口截止时间和关闭任务锁。
- LangGraph 处理完并生成回复后，`send` 节点按换行拆分回复文本，把每段封装成发送任务写入 `sched:send`。
- `SendScheduler` 轮询 `sched:send` 中已到期任务，使用 `ZREM` 认领任务后发送；发送失败会按重试次数重新入队。

### 拟人化如何实现

- 回复生成使用 Agent 自身名称、人设和触发原因作为系统身份约束。
- 用户上下文包含会话类型、群名称、群描述、长期摘要、短期摘要、已知黑话、最近历史消息和当前窗口消息。
- React Agent 可用工具：
    - `web_search`：处理天气、新闻、价格、近期事件等实时信息。
    - `fetch_more_context`：当当前窗口和最近历史不足以理解语境时，继续拉取更早消息。
- Prompt 对表达方式做统一约束：默认中文、短回复、不列点、不讲课、不自称 AI、不重复、不总结聊天记录、优先顺着最新消息接话。

## Window 模块

- 当前窗口：
    - 每个会话独立维护当前窗口消息列表。
    - 进入窗口的消息包含发送者 id、发送者类型、发送者名称、消息类型、文本内容、创建时间和会话序号。
    - 窗口关闭前，新消息继续追加到当前窗口，并刷新截止时间；当前装配默认窗口长度为 10 秒。
- 窗口关闭：
    - 关闭任务先等待 Redis 中记录的截止时间。
    - 如果等待期间截止时间被新消息刷新，关闭任务会继续等待新的截止时间。
    - 到期后读取当前窗口和累计历史，再把当前窗口合并到累计历史。
    - 合并后清理当前窗口缓冲、截止时间和关闭任务锁。
- 累计历史：
    - `all_buf` 保存已关闭窗口的消息，用于后续决策、生成和记忆更新。
    - 长期记忆更新后会裁剪累计历史，超过阈值时保留最后一段近期消息。
- LLM 输入窗口：
    - `decide_reply` 使用完整当前窗口、最近 50 条累计历史、短期摘要和长期摘要。
    - `generate_reply` 使用完整当前窗口、最近 50 条累计历史、短期摘要、长期摘要和最多 8 条相关黑话。
    - `update_short_memory` 使用累计历史加当前窗口后的最近 50 条消息。
    - `update_long_memory` 使用累计历史加当前窗口后的全部消息。
    - `extract_unknown_terms` 只使用当前窗口消息。
    - `infer_glossary_meanings` 使用累计历史加当前窗口后的最近 50 条消息，并结合已有摘要和旧释义。
    - `get_relevant_glossary` 用最近 50 条累计历史加当前窗口拼出的文本做黑话相关性匹配，最多取 8 条进入生成上下文。

## Memory 模块

- 短期记忆：
    - 目标是保留当前仍然有效的话题、安排、共识和分歧。
    - 触发条件是窗口合并次数达到 20 次，或距离上次短期记忆更新超过 30 分钟。
    - 更新输入是旧短期摘要和最近 50 条消息。
    - 更新成功后保存新短期摘要，并重置窗口合并计数。
- 长期记忆：
    - 目标是沉淀群的长期氛围、交流风格、常见话题类型和整体互动方式。
    - 触发条件是累计历史达到 200 条，或距离上次长期记忆更新超过 24 小时。
    - 更新输入是旧长期摘要、会话名称、会话描述和当前累计消息。
    - 更新成功后保存新长期摘要，并裁剪累计历史缓冲；累计历史超过 200 条时保留最后 50 条。
- 黑话词表：
    - 每个窗口都会调用 LLM 从当前消息中抽取可能需要解释的群内黑话候选。
    - 候选词按会话累计出现次数。
    - 词频达到 5 次且尚无释义时，LLM 会结合最近消息、短期摘要、长期摘要推断含义。
    - 已有释义的词从第 5 次出现后，每增加 10 次会重新推断一次，用新上下文修正旧含义。
    - 生成回复时只注入与最近文本相关且已有释义的黑话。
